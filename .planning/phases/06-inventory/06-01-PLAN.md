**Migration (00008_inventory_phase6.sql):**

Follow the exact pattern from `00007_requests_phase4.sql`. The migration must:

1. **ALTER inventory_items table** — add columns:
   - `brand text` (nullable)
   - `model text` (nullable)
   - `serial_number text` (nullable)
   - `acquisition_date date` (nullable) — rename `purchase_date` to `acquisition_date` if cleaner, or add as new column
   - Drop and recreate the `status` CHECK constraint to use values: `('active', 'under_repair', 'broken', 'sold_disposed')` — note the single `sold_disposed` terminal state per CONTEXT.md

2. **ALTER inventory_movements table** — add columns:
   - `receiver_id uuid REFERENCES public.user_profiles(id)` (nullable — the designated receiver)
   - `rejection_reason text` (nullable)
   - `rejected_at timestamptz` (nullable)
   - `cancelled_at timestamptz` (nullable)
   - Drop and recreate the `status` CHECK constraint to use values: `('pending', 'accepted', 'rejected', 'cancelled')`

3. **Create unique partial index** for concurrent transfer guard:
   ```sql
   CREATE UNIQUE INDEX idx_one_pending_movement ON public.inventory_movements (item_id) WHERE status = 'pending' AND deleted_at IS NULL;
   ```

4. **Create `generate_asset_display_id` function** — copy exact pattern from `generate_request_display_id` in 00007, but use entity_type prefix `'asset_'` and display prefix `'AST-'`. Use SECURITY DEFINER SET search_path = public.

5. **Create storage buckets:**
   - `asset-photos` bucket (private, 5MB file size limit, allowed MIME: image/jpeg, image/png, image/webp)
   - `asset-invoices` bucket (private, 10MB file size limit, allowed MIME: image/jpeg, image/png, image/webp, application/pdf)
   - Add RLS policies for both buckets following the `request-photos` bucket pattern (authenticated users can insert to company-scoped paths, select own company's files)

6. **Refine RLS policies on inventory_items:**
   - INSERT restricted to `current_user_role() IN ('ga_staff', 'ga_lead', 'admin')`
   - UPDATE restricted to `current_user_role() IN ('ga_staff', 'ga_lead', 'admin')`
   - SELECT remains company-scoped (all roles can view)

7. **Refine RLS policies on inventory_movements:**
   - INSERT restricted to `current_user_role() IN ('ga_staff', 'ga_lead', 'admin')`
   - UPDATE allowed for: initiator (to cancel), or receiver (to accept/reject), or ga_lead/admin
   - SELECT remains company-scoped

**Server actions (app/actions/asset-actions.ts):**

Follow `app/actions/request-actions.ts` pattern exactly. Use `authActionClient` for all actions, check role inside action body (ga_staff, ga_lead, admin required for mutations).

Create these actions:

1. `createAsset` — schema: assetCreateSchema
   - Role check: ga_staff, ga_lead, admin
   - Call `supabase.rpc('generate_asset_display_id', { p_company_id: profile.company_id })`
   - Insert into inventory_items with all fields (status defaults to 'active')
   - revalidatePath('/inventory')
   - Return { success, assetId, displayId }

2. `updateAsset` — schema: assetEditSchema + asset_id
   - Role check: ga_staff, ga_lead, admin
   - Fetch asset, verify exists and not sold_disposed
   - Update inventory_items fields
   - revalidatePath('/inventory')
   - Return { success }

3. `changeAssetStatus` — schema: assetStatusChangeSchema
   - Role check: ga_staff, ga_lead, admin
   - Fetch current asset, verify exists
   - Validate transition: current status is NOT sold_disposed, new_status is valid target from ASSET_STATUS_TRANSITIONS
   - Update inventory_items.status (and notes if provided)
   - **Auto-pause maintenance:** if new_status is 'broken' or 'sold_disposed', update maintenance_schedules SET is_paused=true, paused_at=now(), paused_reason='Asset status: {new_status}' WHERE item_id=asset_id AND is_paused=false AND deleted_at IS NULL
   - revalidatePath('/inventory')
   - Return { success }

4. `createTransfer` — schema: assetTransferSchema
   - Role check: ga_staff, ga_lead, admin
   - Fetch asset, verify exists, status NOT sold_disposed
   - **Concurrent transfer guard:** check for existing pending movement. The DB unique partial index will also enforce this, but check in code too for a clean error message. Query inventory_movements WHERE item_id=asset_id AND status='pending' AND deleted_at IS NULL. If count > 0, throw "Asset has a pending transfer. Complete or cancel it first."
   - Insert into inventory_movements: company_id, item_id, from_location_id (asset's current location_id), to_location_id, initiated_by (profile.id), receiver_id, status='pending'
   - revalidatePath('/inventory')
   - Return { success, movementId }

5. `acceptTransfer` — schema: transferAcceptSchema
   - Fetch movement, verify exists and status='pending'
   - Verify current user is the designated receiver (movement.receiver_id === profile.id) OR user is ga_lead/admin
   - Update movement: status='accepted', received_by=profile.id, received_at=now()
   - Update inventory_items: location_id = movement.to_location_id
   - revalidatePath('/inventory')
   - Return { success }

6. `rejectTransfer` — schema: transferRejectSchema
   - Fetch movement, verify exists and status='pending'
   - Verify current user is the designated receiver OR ga_lead/admin
   - Update movement: status='rejected', rejection_reason, rejected_at=now()
   - revalidatePath('/inventory')
   - Return { success }

7. `cancelTransfer` — schema: transferCancelSchema
   - Fetch movement, verify exists and status='pending'
   - Verify current user is the initiator (movement.initiated_by === profile.id) OR ga_lead/admin
   - Update movement: status='cancelled', cancelled_at=now()
   - revalidatePath('/inventory')
   - Return { success }

8. `getAssetPhotos` — takes asset_id string
   - Query media_attachments WHERE entity_type IN ('asset_creation', 'asset_status_change') AND entity_id=asset_id
   - Also query for transfer photos: get all movement IDs for this asset, then query media_attachments WHERE entity_type IN ('asset_transfer_send', 'asset_transfer_receive', 'asset_transfer_reject') AND entity_id IN (movementIds)
   - For each photo, generate signed URL via supabase.storage.from('asset-photos').createSignedUrl(file_path, 3600)
   - Return array of { id, entity_type, entity_id, file_name, url, created_at }

9. `getAssetInvoices` — takes asset_id string
   - Query media_attachments WHERE entity_type='asset_invoice' AND entity_id=asset_id
   - For each file, generate signed URL via supabase.storage.from('asset-invoices').createSignedUrl(file_path, 3600)
   - Return array of { id, entity_type, entity_id, file_name, url, created_at }

**Photo upload API route (app/api/uploads/asset-photos/route.ts):**

Copy pattern from `app/api/uploads/request-photos/route.ts` exactly. Changes:
- Accept FormData with: `asset_id` OR `movement_id` (one of the two), `photo_type` (one of: 'creation', 'status_change', 'transfer_send', 'transfer_receive', 'transfer_reject'), and `photos` files
- Map photo_type to entity_type: 'creation' -> 'asset_creation', 'status_change' -> 'asset_status_change', 'transfer_send' -> 'asset_transfer_send', 'transfer_receive' -> 'asset_transfer_receive', 'transfer_reject' -> 'asset_transfer_reject'
- entity_id = asset_id for creation/status_change, movement_id for transfer types
- Upload to `asset-photos` bucket with path: `{company_id}/{entity_id}/{uuid}.{ext}`
- Insert media_attachments rows with correct entity_type and entity_id
- Validate: auth required, max 5 photos, max 5MB each, image MIME types only
- Return 200 with { success: true, count }
