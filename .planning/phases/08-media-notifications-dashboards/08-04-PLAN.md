1. Update `app/actions/request-actions.ts` to trigger notifications after successful mutations:

   a. **triageRequest** — after successful triage update:
      - Fetch the request's `requester_id` and `display_id`
      - Recipients: [requester_id, assigned_to (PIC)] — createNotifications filters out actorId
      - Title: `Request ${display_id} triaged`
      - Body: `Priority: ${priority}, assigned to ${PIC name}` (look up assigned_to name if needed)
      - Type: 'status_change', entityType: 'request', entityId: request.id
      - Use `createNotifications()` from `@/lib/notifications/helpers`
      - IMPORTANT: Do NOT await the notification call. Use `.catch(() => {})` pattern — notification failure must never break triage

   b. **rejectRequest** — after successful rejection:
      - Recipients: [requester_id]
      - Title: `Request ${display_id} rejected`
      - Body: truncated rejection reason (first 100 chars)
      - Type: 'status_change'

   c. **cancelRequest** — after successful cancellation:
      - Fetch GA Lead/Admin users in the same company to notify
      - Query: `select('id').from('user_profiles').in('role', ['ga_lead', 'admin']).eq('company_id', profile.company_id).is('deleted_at', null)`
      - Recipients: ga_lead and admin users
      - Title: `Request ${display_id} cancelled`
      - Type: 'status_change'

   For each action:
   - Import `createNotifications` from `@/lib/notifications/helpers`
   - Fetch the request record with display_id and requester_id before/after the update (depending on what's available)
   - Call `createNotifications()` with `actorId: profile.id` to ensure the acting user is excluded
   - Wrap in try/catch — never let notification creation failure affect the main action's success
