Implement user management page (create, edit, deactivate, reactivate users) with data table and modal forms, and self-service profile drawer with name editing and password change dialog. Wire the profile drawer into the existing user menu.

Purpose: User management allows admins to onboard new team members and control access. The profile drawer gives every user self-service capability to update their name and password without admin intervention. Together, these complete the human side of admin configuration.

Output: User management page at /admin/users with full CRUD, profile Sheet accessible from user menu dropdown, password change dialog.

   export const createUserSchema = z.object({
     email: z.string().email("Valid email is required"),
     full_name: z.string().min(1, "Name is required").max(100),
     role: z.enum(["general_user", "ga_staff", "ga_lead", "finance_approver", "admin"]),
     company_id: z.string().uuid("Company is required"),
     division_id: z.string().uuid("Division is required").optional().or(z.literal("")),
   });

   export const updateUserSchema = z.object({
     full_name: z.string().min(1, "Name is required").max(100),
     role: z.enum(["general_user", "ga_staff", "ga_lead", "finance_approver", "admin"]),
     company_id: z.string().uuid("Company is required"),
     division_id: z.string().uuid("Division is required").optional().or(z.literal("")),
   });

**Server Actions:**

2. Create `app/actions/user-actions.ts` with `'use server'` directive:

   - `createUser`: Uses `adminActionClient`.
     a. Validate with createUserSchema
     b. Create auth user via Supabase Admin API: `supabase.auth.admin.createUser({ email, email_confirm: true })` — this creates the auth.users entry with NO password set and NO invite email sent.
     **IMPORTANT — New user onboarding flow (per user LOCKED decision):** New users created by admin have no password. They must use one of two paths to access the system: (1) Google OAuth login, or (2) the existing "Forgot Password" flow from Phase 2 to set their initial password. The admin should inform the new user of these options out-of-band. No automated invite email is sent.
     c. Insert user_profiles row: `{ id: authUser.id, email, full_name, role, company_id, division_id }`. Set is_active to true (per user decision: immediately active).
     d. Set app_metadata on auth user: `supabase.auth.admin.updateUserById(id, { app_metadata: { role, company_id, division_id } })` — this is read by RLS helper functions.
     e. revalidatePath('/admin/users')
     f. Return `{ success: true, user }`
     g. **Error handling:** If auth user creation fails (e.g., email already exists in auth.users), return error. If profile insert fails, roll back by deleting the auth user.

     **IMPORTANT:** The Supabase Admin API requires the `service_role` key, not the anon key. The `adminActionClient` uses `createClient()` which creates a regular client. For admin operations, create a separate admin Supabase client using `createClient(supabaseUrl, serviceRoleKey)` from `@supabase/supabase-js`. Store `SUPABASE_SERVICE_ROLE_KEY` as a server-only env var (NOT NEXT_PUBLIC_). Create a utility `lib/supabase/admin.ts` that exports an admin client using `createClient` from `@supabase/supabase-js` directly (not the SSR version).

   - `updateUser`: Takes user id + updateUserSchema data.
     a. Update user_profiles row (full_name, role, company_id, division_id)
     b. Update auth user app_metadata: `supabase.auth.admin.updateUserById(id, { app_metadata: { role, company_id, division_id } })`
     c. Per user decision: role changes take effect immediately, no confirmation warning
     d. revalidatePath('/admin/users')

   - `deactivateUser`: Takes user id + optional reason string.
     a. Set deleted_at = now() on user_profiles
     b. Deactivation reason is optional and informational only — do NOT persist it (no deactivation_reason column exists, and notification_preferences is NOT in scope per user decision). The existing middleware deactivation check (deleted_at IS NOT NULL) already blocks access.
     c. Optionally: `supabase.auth.admin.updateUserById(id, { ban_duration: 'none' })` or similar to immediately invalidate sessions — but the middleware already checks deleted_at on every request, so the user will be blocked on next navigation.
     d. revalidatePath('/admin/users')

   - `reactivateUser`: Takes user id.
     a. Set deleted_at = null on user_profiles
     b. Set is_active = true (in case it was also set to false)
     c. revalidatePath('/admin/users')

   - `getUsers`: Fetches user list for the admin page. Since this may need to show deactivated users (when toggle is on), use the admin Supabase client to bypass RLS. Returns users with their division and company names joined.

   Also create `lib/supabase/admin.ts`:
   ```typescript
   import { createClient } from '@supabase/supabase-js';

   export function createAdminClient() {
     const supabaseUrl = process.env.NEXT_PUBLIC_SUPABASE_URL!;
     const serviceRoleKey = process.env.SUPABASE_SERVICE_ROLE_KEY!;
     return createClient(supabaseUrl, serviceRoleKey);
   }
   ```
