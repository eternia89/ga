**Categories (slightly different — global, not company-scoped per user decision):**

4. Create `lib/validations/category-schema.ts`:
   ```typescript
   export const categorySchema = z.object({
     name: z.string().min(1, "Name is required").max(100),
     type: z.enum(["request", "asset"], { required_error: "Type is required" }),
     description: z.string().max(500).optional().or(z.literal("")),
   });
   ```
   **IMPORTANT per user decision:** Categories are "Global/shared across all companies (not company-scoped)". However, the DB schema has a company_id NOT NULL FK. For the UI, the admin should NOT see or select a company for categories. The Server Action should auto-assign the admin's own company_id when creating (since DB requires it), but the table should show categories from ALL companies. In effect, categories behave as if global — they're visible across all companies. The company_id on the DB is a technical artifact that we fill automatically.

5. Create `app/actions/category-actions.ts`:
   - `createCategory`: Auto-fill company_id from the admin's profile (ctx.profile.company_id). Do NOT expose company_id in the form.
   - `deleteCategory`: Check for active requests (category_id), inventory_items (category_id). Return dependency count if > 0.
   - The type field is immutable after creation — update action should not allow changing type

        - Locations tab → `<LocationTable data={locations} companies={companies} />`
        - Categories tab → `<CategoryTable data={categories} />` with sub-tabs for request/asset types
     d. NuqsAdapter wrapping if needed for Next.js App Router

   - Data queries in the server component:
     ```typescript
     // Fetch all companies (including deactivated for admin view)
     const { data: companies } = await supabase
       .from('companies')
       .select('*')
       .order('name');

     // Fetch divisions with company name
     const { data: divisions } = await supabase
       .from('divisions')
       .select('*, company:companies(name)')
       .order('name');

     // Fetch locations with company name
     const { data: locations } = await supabase
       .from('locations')
       .select('*, company:companies(name)')
       .order('name');

     // Fetch all categories (global)
     const { data: categories } = await supabase
       .from('categories')
       .select('*')
       .order('name');
     ```

   **IMPORTANT:** The RLS policies from Phase 1 filter out deleted_at IS NOT NULL on SELECT. To allow admins to see deactivated items when the toggle is on, the server-side query needs to use `.or('deleted_at.is.null,deleted_at.not.is.null')` or a Supabase RPC that bypasses the soft-delete filter. The simplest approach: use a Supabase service role client for admin data fetching, OR adjust the admin SELECT RLS policy in a migration to not filter deleted_at for admin users. **Preferred approach:** Modify the SELECT queries for admin to include a custom RPC or use `.select()` with the Supabase Admin API. Alternatively, fetch all data (including deactivated) by creating a new Supabase function `admin_fetch_all(table_name)` that uses SECURITY DEFINER. **Simplest approach for now:** The DataTable shows deactivated items only when the toggle is on. Since RLS filters them out, add a separate query to fetch deactivated items when the toggle is active. Use two queries: one for active (default RLS), one for deactivated (using an RPC or service role). OR: Create a new migration that adds admin-override SELECT policies that allow admin to see deleted_at IS NOT NULL records.

   **Recommended approach:** Create a small Supabase migration (00007) that adds a new SELECT policy for each config table allowing admin to see soft-deleted records:
   ```sql
   CREATE POLICY "companies_select_admin_all" ON public.companies
     FOR SELECT TO authenticated
     USING (current_user_role() = 'admin');
   ```
   This policy (combined with the existing policy via OR) allows admin to see all records including soft-deleted ones. Non-admin users still only see active records via the original policy.
