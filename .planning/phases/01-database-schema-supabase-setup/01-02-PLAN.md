    Create three SECURITY DEFINER + STABLE functions:

    ```sql
    -- Extract company_id from JWT app_metadata
    CREATE OR REPLACE FUNCTION auth.user_company_id()
    RETURNS uuid
    LANGUAGE sql
    STABLE
    AS $$
      SELECT COALESCE(
        (auth.jwt() -> 'app_metadata' ->> 'company_id')::uuid,
        '00000000-0000-0000-0000-000000000000'::uuid
      );
    $$;

    -- Extract division_id from JWT app_metadata
    CREATE OR REPLACE FUNCTION auth.user_division_id()
    RETURNS uuid
    LANGUAGE sql
    STABLE
    AS $$
      SELECT (auth.jwt() -> 'app_metadata' ->> 'division_id')::uuid;
    $$;

    -- Extract user role from JWT app_metadata
    CREATE OR REPLACE FUNCTION auth.user_role()
    RETURNS text
    LANGUAGE sql
    STABLE
    AS $$
      SELECT COALESCE(
        auth.jwt() -> 'app_metadata' ->> 'role',
        'general_user'
      );
    $$;
    ```

<task type="auto">
  <name>Task 2: Create baseline RLS policies for all tables</name>
  <files>
    supabase/migrations/00003_rls_policies.sql
  </files>
  <action>
    Create a migration with baseline RLS policies for every table. Each table gets four operation policies (SELECT, INSERT, UPDATE, DELETE) following these rules:

    **Policy patterns:**

    - **SELECT** (read): `company_id = auth.user_company_id() AND deleted_at IS NULL`
    - **INSERT** (create): `company_id = auth.user_company_id()`
    - **UPDATE** (modify): USING `company_id = auth.user_company_id()` (NO deleted_at check — allows soft-delete), WITH CHECK `company_id = auth.user_company_id()`
    - **DELETE** (hard delete): Deny all via restrictive policy — soft delete only (no policy = no permission for hard delete)

    **Critical: Soft-delete UPDATE pattern**
    The UPDATE policy must NOT include `deleted_at IS NULL` in the USING clause. This is because when a user sets `deleted_at = now()` (soft delete), PostgreSQL checks the new row against the policy. If the SELECT policy includes `deleted_at IS NULL`, the update would fail. By omitting the deleted_at check from UPDATE, soft-delete operations succeed.

    **Per-table policies:**

    All policies apply `TO authenticated` role.

    **1. companies**
    ```sql
    CREATE POLICY "companies_select" ON public.companies
      FOR SELECT TO authenticated
      USING (id = auth.user_company_id() AND deleted_at IS NULL);

    CREATE POLICY "companies_insert" ON public.companies
      FOR INSERT TO authenticated
      WITH CHECK (id = auth.user_company_id());

    CREATE POLICY "companies_update" ON public.companies
      FOR UPDATE TO authenticated
      USING (id = auth.user_company_id())
      WITH CHECK (id = auth.user_company_id());
    ```
    Note: Companies use `id = auth.user_company_id()` (not company_id, since companies IS the company).

    **2-5. divisions, locations, categories, user_profiles** — Standard pattern with `company_id = auth.user_company_id()`.

    **6. id_counters** — SELECT and UPDATE only (no direct INSERT/DELETE from client; managed by generate_display_id function).
    ```sql
    CREATE POLICY "id_counters_select" ON public.id_counters
      FOR SELECT TO authenticated
      USING (company_id = auth.user_company_id());

    CREATE POLICY "id_counters_update" ON public.id_counters
      FOR UPDATE TO authenticated
      USING (company_id = auth.user_company_id())
      WITH CHECK (company_id = auth.user_company_id());
    ```

    **7. requests** — Standard pattern. SELECT includes `deleted_at IS NULL`.

    **8. jobs** — Standard pattern. SELECT includes `deleted_at IS NULL`.

    **9. job_comments** — Standard pattern.

    **10. inventory_items** — Standard pattern.

    **11. inventory_movements** — Standard pattern.

    **12. maintenance_templates** — Standard pattern.

    **13. maintenance_schedules** — Standard pattern.

    **14. notifications** — User-scoped: `user_id = auth.uid() AND company_id = auth.user_company_id()` for SELECT. INSERT uses company_id check. UPDATE (mark as read) uses user_id + company_id.
    ```sql
    CREATE POLICY "notifications_select" ON public.notifications
      FOR SELECT TO authenticated
      USING (user_id = auth.uid() AND deleted_at IS NULL);

    CREATE POLICY "notifications_update" ON public.notifications
      FOR UPDATE TO authenticated
      USING (user_id = auth.uid())
      WITH CHECK (user_id = auth.uid());
    ```
    Note: Notifications INSERT will be done via service_role (server-side triggers), so no INSERT policy needed for authenticated users. Add a service-role-only insert pattern comment.

    **15. media_attachments** — Standard company_id pattern.

    **16. audit_logs** — Read-only for now (admin viewer comes in Phase 9). No write policies — writes happen via SECURITY DEFINER trigger.
    NOTE: audit_logs is an immutable table with NO deleted_at column (see Plan 01-01). Therefore, the SELECT policy intentionally omits the `deleted_at IS NULL` filter — this is correct, not an oversight.
    ```sql
    CREATE POLICY "audit_logs_select" ON public.audit_logs
      FOR SELECT TO authenticated
      USING (company_id = auth.user_company_id());
    ```

    **Naming convention:** `{table}_{operation}` (e.g., `requests_select`, `requests_insert`, `requests_update`).

    **Important notes:**
    - These are BASELINE policies. Role-specific policies (e.g., GA Lead can see all divisions, General User only sees own division's requests) will be refined in Phase 2 (RBAC). These baseline policies ensure company isolation and soft-delete filtering from day one.
    - No DELETE policies means hard deletes are impossible through the API — only soft deletes via UPDATE.
    - The service_role key bypasses RLS entirely, so server-side admin operations still work.
  </action>

  <action>
    Create a migration that establishes the generic audit trigger function and attaches it to all domain tables.

    **Part A: Generic audit trigger function**

    ```sql
    CREATE OR REPLACE FUNCTION public.audit_trigger()
    RETURNS TRIGGER
    LANGUAGE plpgsql
    SECURITY DEFINER  -- Bypasses RLS to write to audit_logs
    SET search_path = public
    AS $$
    DECLARE
      v_old_data jsonb;
      v_new_data jsonb;
      v_changed_fields text[];
      v_record_id uuid;
      v_company_id uuid;
    BEGIN
      -- Determine record ID and build JSONB
      IF TG_OP = 'DELETE' THEN
        v_old_data := to_jsonb(OLD);
        v_new_data := NULL;
        v_record_id := OLD.id;
        v_company_id := OLD.company_id;
      ELSIF TG_OP = 'UPDATE' THEN
        v_old_data := to_jsonb(OLD);
        v_new_data := to_jsonb(NEW);
        v_record_id := NEW.id;
        v_company_id := NEW.company_id;
        -- Compute changed fields
        SELECT ARRAY_AGG(key) INTO v_changed_fields
        FROM jsonb_each(v_old_data) AS o(key, value)
        WHERE v_old_data->key IS DISTINCT FROM v_new_data->key;
      ELSIF TG_OP = 'INSERT' THEN
        v_old_data := NULL;
        v_new_data := to_jsonb(NEW);
        v_record_id := NEW.id;
        v_company_id := NEW.company_id;
      END IF;

      -- Insert audit record
      INSERT INTO public.audit_logs (
        id,
        company_id,
        table_name,
        record_id,
        operation,
        old_data,
        new_data,
        changed_fields,
        user_id,
        user_email,
        performed_at
      ) VALUES (
        gen_random_uuid(),
        v_company_id,
        TG_TABLE_NAME,
        v_record_id,
        TG_OP,
        v_old_data,
        v_new_data,
        v_changed_fields,
        auth.uid(),
        COALESCE(auth.jwt() ->> 'email', 'system'),
        now()
      );

      RETURN COALESCE(NEW, OLD);
    END;
    $$;
    ```

    Note on SECURITY DEFINER: This is required because audit_logs has RLS enabled, and the trigger runs in the context of the user making the change. SECURITY DEFINER makes the trigger run as the function owner (postgres superuser), bypassing RLS on the audit_logs table. The `SET search_path = public` prevents search_path hijacking (a security best practice for SECURITY DEFINER functions).

    **Part B: Handle tables without company_id**

    The `companies` table does not have a `company_id` column — it IS the company. For the audit trigger on companies, use `OLD.id` or `NEW.id` as the company_id:

    Create a slightly modified trigger for the companies table:
    ```sql
    CREATE OR REPLACE FUNCTION public.audit_trigger_companies()
    RETURNS TRIGGER
    LANGUAGE plpgsql
    SECURITY DEFINER
    SET search_path = public
    AS $$
    DECLARE
      v_old_data jsonb;
      v_new_data jsonb;
      v_changed_fields text[];
      v_record_id uuid;
    BEGIN
      IF TG_OP = 'DELETE' THEN
        v_old_data := to_jsonb(OLD);
        v_record_id := OLD.id;
      ELSIF TG_OP = 'UPDATE' THEN
        v_old_data := to_jsonb(OLD);
        v_new_data := to_jsonb(NEW);
        v_record_id := NEW.id;
        SELECT ARRAY_AGG(key) INTO v_changed_fields
        FROM jsonb_each(v_old_data) AS o(key, value)
        WHERE v_old_data->key IS DISTINCT FROM v_new_data->key;
      ELSIF TG_OP = 'INSERT' THEN
        v_new_data := to_jsonb(NEW);
        v_record_id := NEW.id;
      END IF;

      INSERT INTO public.audit_logs (
        id, company_id, table_name, record_id, operation,
        old_data, new_data, changed_fields, user_id, user_email, performed_at
      ) VALUES (
        gen_random_uuid(),
        COALESCE(NEW.id, OLD.id),  -- company IS the record
        TG_TABLE_NAME, v_record_id, TG_OP,
        v_old_data, v_new_data, v_changed_fields,
        auth.uid(), COALESCE(auth.jwt() ->> 'email', 'system'), now()
      );

      RETURN COALESCE(NEW, OLD);
    END;
    $$;
    ```
