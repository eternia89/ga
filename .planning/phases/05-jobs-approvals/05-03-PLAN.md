**A. Create `app/(dashboard)/jobs/[id]/page.tsx`** — server component:
- Fetch job with all relations using Promise.all (same pattern as requests/[id]/page.tsx):
  1. Job data: `jobs` with location(name), category(name), pic (user_profiles!assigned_to full_name), created_by_user (user_profiles!created_by full_name), job_requests with nested request(id, display_id, title, status, requester via user_profiles!requester_id full_name)
  2. Audit logs: `audit_logs` where table_name='jobs' and record_id=id, ordered by performed_at ascending
  3. Job comments: `job_comments` where job_id=id and deleted_at IS NULL, with user(full_name), ordered by created_at ascending
  4. Comment photos: `media_attachments` where entity_type='job_comment' and entity_id IN (comment ids)
- Process audit logs into timeline events using the same classification pattern from request detail:
  - Rejection check: if new_values has approval_rejection_reason -> 'approval_rejection' event
  - Cancellation: if new_values.status === 'cancelled' -> 'cancellation' event
  - Assignment: if new_values.assigned_to changed -> 'assignment' event (resolve user name)
  - Approval submitted: if new_values.status === 'pending_approval' -> 'approval_submitted' event
  - Approval: if new_values.status changed and approved_at set -> 'approval' event
  - Status change: if new_values.status changed -> 'status_change' event
  - Field update: anything else -> 'field_update' event
- Resolve performer names from user_profiles for audit log entries.
- Render breadcrumb: Jobs > {display_id}.
- Pass processed data to `<JobDetailClient />`.

**B. Create `components/jobs/job-detail-client.tsx`** — client wrapper:
- Same pattern as RequestDetailClient. Manages edit state.
- Props: job, timelineEvents, comments, commentPhotos, currentUser, referenceData (locations, categories, users for edit form).
- Two-column layout: `grid grid-cols-[1fr_380px] max-lg:grid-cols-1 gap-6` (desktop-first per CLAUDE.md).
- Left column: `<JobDetailInfo />` and `<JobDetailActions />`.
- Right column: `<JobTimeline />` and `<JobCommentForm />` (if user can comment).

**C. Create `components/jobs/job-detail-info.tsx`**:
- Display all job fields in a structured card layout:
  - Header: display_id, title, JobStatusBadge
  - Fields: PIC (name or "Unassigned"), Category, Location, Priority (JobPriorityBadge), Description
  - **Estimated Cost**: formatted with IDR (`new Intl.NumberFormat('id-ID', { style: 'currency', currency: 'IDR', minimumFractionDigits: 0 }).format(cost)`) — display prominently
  - **Linked Requests**: show as compact inline previews per CONTEXT.md locked decision. Each preview shows: request display_id (clickable Link to /requests/[id]), title (truncated), status badge, requester name. Use a compact card or list-item style.
  - Dates: created_at, assigned date (if assigned), started_at, completed_at — all formatted dd-MM-yyyy, HH:mm:ss
  - If approval was submitted: show approval_submitted_at date
  - If approved: show approved_at, approved_by
  - If rejected: show rejection reason, approval_rejected_at, rejection by whom
- When in edit mode: render `<JobEditForm />` (reuse job-form.tsx in edit mode — but that's a future concern; for now, just show info).

**D. Create `components/jobs/job-detail-actions.tsx`**:
- Context-sensitive action buttons based on user role and job status (per CONTEXT.md):
  - **Created** status:
    - GA Lead/Admin: "Assign" button (opens assign dialog with PIC combobox) + "Edit" + "Cancel"
  - **Assigned** status:
    - GA Lead/Admin OR PIC: "Start Work" button (transitions to in_progress)
    - GA Lead/Admin: "Submit for Approval" button (only shown if estimated_cost exists and cost >= budget threshold) + "Edit" + "Reassign" + "Cancel"
  - **Pending Approval** status:
    - Finance Approver/Admin: "Approve" + "Reject" buttons (reject opens dialog requiring reason)
    - Others: read-only "Awaiting CEO Approval" indicator
  - **In Progress** status:
    - GA Lead/Admin OR PIC: "Mark Complete" button
    - GA Lead/Admin: "Cancel"
  - **Completed/Cancelled**: no action buttons (read-only)
- Use AlertDialog for cancel confirmation. Use Dialog for assignment (Combobox for PIC selection) and rejection (Textarea for reason).
- Each button calls the appropriate server action (updateJobStatus, cancelJob, assignJob, submitForApproval, approveJob, rejectJob).
- Use InlineFeedback for success/error messages (persistent, user-dismissable).
