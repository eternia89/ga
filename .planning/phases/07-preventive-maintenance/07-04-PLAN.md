**PM job actions (app/actions/pm-job-actions.ts):**
Use `authActionClient` pattern.

1. **`savePMChecklistItem`** — save-as-you-go for individual checklist items (per CONTEXT.md: "PIC fills it out item by item with save-as-you-go").
   - Schema: z.object({ jobId: z.string().uuid(), itemId: z.string(), value: z.union([z.boolean(), z.string(), z.number(), z.null()]) })
   - Fetch job's current checklist_responses JSONB
   - Find the item by item_id in the responses array, update its value and set completed_at to current ISO timestamp
   - Update the job's checklist_responses column with the modified array
   - Return { success: true, completedCount, totalCount } so UI can show progress

2. **`savePMChecklistPhoto`** — handle photo-type checklist items separately since they may involve file upload.
   - Schema: z.object({ jobId: z.string().uuid(), itemId: z.string(), photoUrls: z.array(z.string()) })
   - Same fetch-modify-replace pattern for checklist_responses
   - Update item value to the array of photo URLs
   - NOTE: The actual file upload goes through the existing job photos API route. This action just saves the URLs to the checklist response.

3. **`completePMChecklist`** — called when all checklist items are filled. Validates that all items have non-null values (all items are required per CONTEXT.md: "All checklist items are required when filling out a PM job").
   - Returns error if any items have null value
   - If all complete: does NOT change job status (PIC still marks job as complete via normal job status change flow)
   - But does update a `checklist_completed_at` timestamp in the checklist_responses metadata

4. **`advanceFloatingSchedule`** — called when a PM job is marked as completed (integrate into the existing job completion flow from Phase 5).
   - Fetch the job to get maintenance_schedule_id
   - If maintenance_schedule_id exists AND schedule.interval_type = 'floating':
     - Update maintenance_schedules: set next_due_at = now() + interval_days, set last_completed_at = now()
   - If interval_type = 'fixed': only update last_completed_at (next_due_at was already advanced by the cron)
   - Per RESEARCH.md Pitfall 2: floating next_due_at is ONLY updated on completion, never by the cron

**IMPORTANT:** The Phase 5 job completion action will need to call advanceFloatingSchedule after marking a PM job as completed. Add a TODO comment in the action noting this integration point. The executor should wire this into the existing job-actions.ts `completeJob` action (or document the hook point clearly in the SUMMARY).

**PM checklist (components/maintenance/pm-checklist.tsx):**
Client component rendered on the PM job detail page (inside the job detail from Phase 5). Shows the checklist items from the job's `checklist_responses` JSONB.

Props: `{ jobId: string, checklist: PMJobChecklist, jobStatus: string, canEdit: boolean }`
- `canEdit` is true when job status is 'assigned' or 'in_progress' and current user is the assigned PIC
- Header: template name (from checklist.template_name), progress indicator (e.g., "3/8 completed")
- Per CONTEXT.md: "Checklist appears inline on the PM job detail page; PIC fills it out item by item with save-as-you-go"
- Renders each item via PMChecklistItem component
- Progress bar showing completion percentage (Claude's discretion on design)
- If all items completed, show a success state (green check + "All items completed")
- If job is completed/cancelled, show read-only view (no edit controls)

**PM checklist item (components/maintenance/pm-checklist-item.tsx):**
Renders a single checklist item with type-appropriate input control.

Props: `{ item: ChecklistResponse, jobId: string, canEdit: boolean, onSaved: () => void }`

Type-specific controls:
1. **checkbox**: Checkbox component. Checked = completed.
2. **pass_fail**: Two radio buttons or toggle: "Pass" (green) / "Fail" (red). Clear visual distinction.
3. **numeric**: Number Input. Shows unit suffix if present in the item definition.
4. **text**: Textarea (max 1000 chars per CLAUDE.md long text convention).
5. **photo**: Upload area (reuse existing photo upload pattern from Phase 4/5). Shows thumbnail previews of uploaded photos.
6. **dropdown**: Select component with the predefined options from the checklist item definition.

Save behavior:
- Each control has a debounced auto-save (e.g., 500ms debounce for text/numeric, immediate for checkbox/pass_fail/dropdown)
- Calls savePMChecklistItem server action
- Shows subtle save indicator (check icon or "Saved" text that fades)
- For photo type: calls savePMChecklistPhoto after upload completes

Read-only mode (canEdit=false):
- Show values as static text/badges instead of input controls
- Checkbox: check/X icon
- Pass/fail: colored badge
- Numeric: value + unit
- Text: plain text
- Photo: thumbnails (clickable for lightbox)
- Dropdown: selected option as text

**Overdue badge (components/maintenance/overdue-badge.tsx):**
Simple presentational component per CONTEXT.md: "Overdue PM jobs: red 'Overdue' badge on the job list and job detail page header."

Props: `{ nextDueAt: string | null, jobStatus: string }`

  - nextDueAt is in the past (nextDueAt < now())
