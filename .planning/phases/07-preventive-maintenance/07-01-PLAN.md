**Migration (00009_pm_phase7.sql):**
1. Add `checklist_responses jsonb DEFAULT NULL` column to `public.jobs` table — stores the full checklist snapshot (template definition + responses) for PM jobs so completed jobs are immune to template edits.
2. Add `is_active boolean DEFAULT true` column to `public.maintenance_schedules` — tracks manual deactivation independently from auto-pause (`is_paused`) and soft-delete (`deleted_at`). Per CONTEXT.md: "GA Lead can manually activate/deactivate schedules (independent of asset status auto-pause)".
3. Create unique partial index on `jobs(maintenance_schedule_id) WHERE deleted_at IS NULL AND status NOT IN ('completed', 'cancelled') AND maintenance_schedule_id IS NOT NULL` — prevents duplicate open PM jobs per schedule (REQ-PM-009).
4. Create index `idx_jobs_pm_type ON jobs(job_type, status) WHERE deleted_at IS NULL AND job_type = 'preventive_maintenance'` — optimizes overdue PM job queries.
5. Create `generate_pm_jobs()` SECURITY DEFINER function (see RESEARCH.md Pattern 3) that:
   - Queries maintenance_schedules joined with maintenance_templates and inventory_items
   - Filters: deleted_at IS NULL, is_paused = false, is_active = true, mt.is_active = true, next_due_at <= now()
   - Deduplication: NOT EXISTS open job from same schedule_id
   - For each due schedule: generates display_id via generate_display_id(), inserts job with job_type='preventive_maintenance', title='[Template Name] - [Asset Name]', status='created', created_by = first ga_lead in company
   - Snapshots the checklist into the new job's checklist_responses as `{template_name, template_id, items: [...itemsWithNullValues]}`
   - For FIXED schedules: advances next_due_at by interval_days. For FLOATING: does NOT update next_due_at (handled on job completion)
6. Schedule the function with pg_cron: `SELECT cron.schedule('generate-pm-jobs', '5 0 * * *', 'SELECT generate_pm_jobs()')` — runs daily at 00:05 UTC.
7. Refine RLS policies for maintenance_templates and maintenance_schedules:
   - DROP existing INSERT/UPDATE policies if any
   - CREATE INSERT policy: company_id match + role IN ('ga_lead', 'admin')
   - CREATE UPDATE policy: company_id match + role IN ('ga_lead', 'admin')
   - SELECT policies already exist from Phase 1 (company isolation + soft-delete)

**TypeScript types (lib/types/maintenance.ts):**
Define a discriminated union for checklist items following RESEARCH.md Pattern 1:
- `ChecklistItemBase`: { id: string (UUID), label: string, sort_order: number }
- 6 concrete types: CheckboxItem, PassFailItem, NumericItem (with optional unit: string), TextItem, PhotoItem, DropdownItem (with options: string[])
- `ChecklistItem = CheckboxItem | PassFailItem | ...` discriminated on `type` field
- `ChecklistResponse`: { item_id: string, type: ChecklistItem['type'], label: string, value: boolean | 'pass' | 'fail' | number | string | null, photo_urls?: string[], completed_at?: string }
- `PMJobChecklist`: { template_name: string, template_id: string, items: ChecklistResponse[] }
- `MaintenanceTemplate`: { id, company_id, category_id, name, description, checklist: ChecklistItem[], is_active, deleted_at, created_at, updated_at } with optional joined `category?: { name: string, type: string }`
- `MaintenanceSchedule`: { id, company_id, item_id, template_id, assigned_to, interval_days, interval_type: 'fixed' | 'floating', last_completed_at, next_due_at, is_paused, paused_at, paused_reason, is_active, deleted_at, created_at, updated_at } with optional joined relations: `template?: { name: string, checklist: ChecklistItem[] }`, `asset?: { name: string, display_id: string }`, `category?: { name: string }`
- `ScheduleDisplayStatus = 'active' | 'paused_auto' | 'paused_manual' | 'deactivated'`

**Constants (lib/constants/checklist-types.ts):**
- `CHECKLIST_TYPES` object mapping type keys to labels: { checkbox: 'Checkbox', pass_fail: 'Pass/Fail', numeric: 'Numeric', text: 'Text', photo: 'Photo', dropdown: 'Dropdown' }
- `CHECKLIST_TYPE_ICONS` mapping type keys to icon strings or emojis for the template builder

- `SCHEDULE_STATUS_COLORS`: badge color classes for each status (green, amber, yellow, gray)

**Template validation (lib/validations/template-schema.ts):**
Follow RESEARCH.md code example. Create Zod schemas:
- `checklistItemBase`: z.object({ id: z.string().uuid(), label: z.string().min(1).max(200), sort_order: z.number().int().min(0) })
- 6 type-specific schemas extending base: checkboxItem, passFailItem, numericItem (+ optional unit: z.string().max(20)), textItem, photoItem, dropdownItem (+ options: z.array(z.string().max(100)).min(1).max(20))
- `checklistItemSchema`: z.discriminatedUnion('type', [...all 6])
- `templateCreateSchema`: z.object({ name: z.string().min(1).max(100), description: z.string().max(200).optional(), category_id: z.string().uuid(), checklist: z.array(checklistItemSchema).min(1) })
- `templateEditSchema`: same as create (templates can be freely edited per CONTEXT.md)
- Export types: ChecklistItemFormData, TemplateCreateFormData

**Schedule validation (lib/validations/schedule-schema.ts):**
- `scheduleCreateSchema`: z.object({ template_id: z.string().uuid(), item_id: z.string().uuid(), interval_days: z.number().int().min(1).max(365), interval_type: z.enum(['fixed', 'floating']).default('floating'), start_date: z.string().optional() })
- Export type: ScheduleCreateFormData

**Template server actions (app/actions/template-actions.ts):**
Use `authActionClient` pattern from existing actions (see request-actions.ts for pattern). All actions check `role IN ('ga_lead', 'admin')` inside the action body (same pattern as request triage actions).

1. `createTemplate(data)` — validates with templateCreateSchema, inserts into maintenance_templates with company_id from user context, returns { id, display_id: null }. Ensure category_id references an 'asset' type category (maintenance is for assets).
2. `updateTemplate(id, data)` — validates with templateEditSchema, updates name/description/category_id/checklist. Per CONTEXT.md: "Templates can be edited freely; future auto-generated PM jobs use the latest version; completed jobs keep their original checklist."
3. `deactivateTemplate(id)` — sets is_active = false. Per CONTEXT.md: "Templates can be deactivated but not deleted; prevents orphaned schedules." Check that no active schedules reference this template before deactivating — if active schedules exist, return error with count.
4. `reactivateTemplate(id)` — sets is_active = true.
5. `getTemplates()` — fetches all templates for user's company where deleted_at IS NULL, joins category name, returns sorted by name. Include item_count computed from checklist JSONB array length.
6. `getTemplateById(id)` — fetches single template with category join.

**Schedule server actions (app/actions/schedule-actions.ts):**
1. `createSchedule(data)` — validates with scheduleCreateSchema. IMPORTANT: validate that the template's category_id matches the asset's category_id (prevent category mismatch per RESEARCH.md Pitfall 7). Calculate initial next_due_at: if start_date provided use it, else now() + interval_days. Insert into maintenance_schedules.
2. `updateSchedule(id, data)` — update interval_days, interval_type. Recalculate next_due_at if interval changes.
3. `deactivateSchedule(id)` — set is_active = false. Per CONTEXT.md: "GA Lead can manually activate/deactivate schedules." Also cancel any open PM jobs linked to this schedule (status IN created, assigned, in_progress).
4. `activateSchedule(id)` — set is_active = true, recalculate next_due_at = now() + interval_days.
5. `deleteSchedule(id)` — soft delete (set deleted_at). Historical PM jobs remain per CONTEXT.md.
6. `getSchedules()` — fetch all schedules for company, join template name, asset name/display_id. Include derived display status via helper function.
7. `getSchedulesByAssetId(assetId)` — fetch schedules for a specific asset (for asset detail page "Maintenance Schedules" section).
8. `pauseSchedulesForAsset(assetId, reason)` — auto-pause helper: pauses all active schedules for asset, cancels open PM jobs. Called from asset status change action. Uses `paused_reason: 'auto:asset_{status}'` prefix convention.
9. `resumeSchedulesForAsset(assetId)` — auto-resume helper: resumes auto-paused schedules, recalculates next_due_at = now() + interval_days (fresh start per CONTEXT.md). Only resumes schedules where paused_reason starts with 'auto:'.
10. `deactivateSchedulesForAsset(assetId)` — permanent deactivation for sold/disposed assets: soft-deletes all schedules. Called when asset reaches terminal status.
