**2. Create `lib/supabase/server.ts`** -- Server-side Supabase client:
- Export async function `createClient()` that uses `createServerClient` from `@supabase/ssr`
- Read cookies via `cookies()` from `next/headers`
- Use `process.env.NEXT_PUBLIC_SUPABASE_URL!` and `process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!`

- Follow exact pattern from research doc (Pattern 1)

**3. Create `lib/supabase/client.ts`** -- Browser-side Supabase client:
- Export function `createClient()` that uses `createBrowserClient` from `@supabase/ssr`
- Use same env vars as server client
- Follow exact pattern from research doc (Pattern 3)

**4. Create `lib/supabase/middleware.ts`** -- Middleware Supabase client:
- Export async function `updateSession(request: NextRequest)` that:
  - Creates `NextResponse.next()` with forwarded headers
  - Creates Supabase server client with cookie bridge (read from request, write to both request and response)
  - Calls `supabase.auth.getUser()` to refresh session (CRITICAL: this refreshes expired tokens)
  - Returns `{ response, user }` tuple (not just response -- middleware.ts needs user for redirect logic)
- Follow Pattern 2 from research doc but return user as well

**5. Create `middleware.ts`** (root level) -- Auth middleware:
- Import `updateSession` from `@/lib/supabase/middleware`
- Import `createServerClient` from `@supabase/ssr` for deactivation check
- Logic flow:
  1. Call `updateSession(request)` to get response + user
  2. Define route categories:
     - Auth routes: `/login`, `/reset-password`, `/update-password`
     - Public routes: auth routes + `/api/auth/callback`
     - Everything else = protected
  3. If user is NOT authenticated AND route is protected:
     - Redirect to `/login?redirect={encodeURIComponent(request.nextUrl.pathname + request.nextUrl.search)}`
  4. If user IS authenticated AND route is an auth route (login, reset-password):
     - Redirect to `/` (dashboard) -- already logged in, don't show login page
  5. If user IS authenticated AND route is protected:
     - Fetch user profile from `user_profiles` table to check `deleted_at` (deactivated)
     - Use the Supabase client from updateSession (or create a new server client from the middleware request)
     - If profile has `deleted_at IS NOT NULL` (deactivated):
       - Sign out the user: `await supabase.auth.signOut()`
       - Redirect to `/login?error=deactivated`
     - NOTE: This profile check should be lightweight. Cache strategy: only check on page navigations, not on every static asset request.
  6. Return response
- Export `config.matcher`: `['/((?!_next/static|_next/image|favicon.ico|.*\\.(?:svg|png|jpg|jpeg|gif|webp)$).*)']`
- Per user decision: 30-day session is controlled by Supabase session settings (not middleware). Middleware just refreshes tokens.

**6. Create `app/api/auth/callback/route.ts`** -- OAuth callback:
- Export async `GET` handler
- Extract `code` from URL searchParams
- Extract `next` param (return URL) defaulting to `/`
- If code exists: create server Supabase client, call `exchangeCodeForSession(code)`
- After exchange, check if user has a profile in `user_profiles` table
  - If NO profile found (Google email not matching any admin-created account):
    - Sign out the user immediately
    - Redirect to `/login?error=no_account`
  - If profile has `deleted_at IS NOT NULL`:
    - Sign out the user
    - Redirect to `/login?error=deactivated`
  - Otherwise: redirect to `next` URL
- Handle errors: redirect to `/login?error=auth_callback_failed`

**7. Create `app/api/auth/signout/route.ts`** -- Server-side sign out:
- Export async `POST` handler
- Create server Supabase client
- Call `supabase.auth.signOut({ scope: 'global' })` (signs out all sessions per user decision)
- Redirect to `/login`

**Important implementation notes:**
- Do NOT use `supabase.auth.getSession()` for user verification -- always use `supabase.auth.getUser()` (server-side validation, not just JWT decode)
- The middleware deactivation check queries `user_profiles` -- this adds a DB call per request. To minimize impact, only run this check on document requests (not API/static). The matcher already excludes static files.
- Rate limiting on auth (REQ-AUTH-006): Supabase has built-in rate limiting (60 req/min per IP for auth endpoints). This fulfills REQ-AUTH-006. Per user decision: "Progressive rate limiting (slow down responses), never full lockout." Supabase's built-in limiting handles this transparently — repeated failed logins get progressively slower responses but the user is never fully locked out. No custom rate limiting middleware needed.
- Add a code comment in the login page error handler documenting that rate limit errors from Supabase (HTTP 429) should display: "Too many attempts. Please wait a moment." This ensures the progressive limiting is surfaced to users correctly.
