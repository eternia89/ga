## 2. RLS Policy Completeness

**Status:** Fixed
**Scope:** All migration RLS policies vs app query patterns

### Findings

| # | Severity | Table | Issue | Action |
|---|----------|-------|-------|--------|
| 1 | CRITICAL | company_settings | SELECT/INSERT/UPDATE policies hardcoded to primary company only — budget threshold lookup fails for secondary company jobs | **Fix:** new migration expanding RLS for multi-company |
| 2 | HIGH | job-actions.ts:131 | Budget threshold query uses `profile.company_id` instead of effectiveCompanyId | **Fix:** use effectiveCompanyId |
| 3 | CRITICAL | maintenance_schedules | INSERT/UPDATE missing multi-company | **Skip:** all writes use adminSupabase (service role bypasses RLS) |
| 4 | HIGH | id_counters | Missing multi-company in SELECT/UPDATE | **Skip:** all access via SECURITY DEFINER RPCs |
| 5 | MEDIUM | job_requests | No UPDATE policy | **Skip:** no UPDATE mutations exist |
| 6 | LOW | user_company_access | No INSERT/UPDATE/DELETE policies | **Skip:** service role only by design |

### Key Insight
Most "missing" multi-company policies are theoretical because the app uses `adminSupabase` (service role) for writes. The one REAL bug is company_settings — it's queried via user-scoped Supabase for budget threshold checks.
